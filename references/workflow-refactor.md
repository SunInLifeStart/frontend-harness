# refactor: 代码重构

## 用法

```
refactor: <重构目标>
refactor: [截图] <重构目标>
```

不改变外部行为，只改内部结构。重构前后功能等价。

> **执行模式说明：** 本工作流中所有「调用 references/xxx.md」均指**读取对应文件，按其中步骤顺序执行**。所有步骤由当前 Agent 顺序完成，不启动子 Agent。

---

## 阶段 0: 环境预检

1. **读取环境配置** — 检查 `.harness-env.json` 是否存在：
   - 不存在 → 调用 `references/framework-detector.md` 执行环境探测
   - 存在 → 读取缓存

---

## 阶段 1: 理解重构意图

### 1.1 解析输入

识别重构类型：
- **结构重构**：拆分大组件、提取公共逻辑、调整目录结构
- **性能重构**：优化渲染、减少重复计算、懒加载
- **规范对齐**：Options API → Composition API、class → function 等
- **依赖清理**：移除未使用依赖、统一工具函数

### 1.2 定位代码

根据重构目标搜索相关代码，定位到具体文件和代码位置。

### 1.3 影响分析

列出所有受影响文件和调用链。

**影响 >5 个文件 → ⏸️ 停下等确认**

---

## 阶段 2: 重构方案

### 2.1 加载框架规范

根据 `.harness-env.json` 中的 `framework` 字段加载对应规范：
- Vue → 读取 `references/vue-best-practices.md`
- React → 读取 `references/react-best-practices.md`

### 2.2 扫描资产

读取 `references/component-scanner.md`，**按需扫描**（scope: targeted）：
- 扫描与重构目标相关的模块

### 2.3 制定方案

- 每步必须原子化（可独立验证）
- 标注每步的功能等价性验证方式
- 标注是否需要中间 commit

### ⏸️ 审批门禁

**影响 >3 个文件时，输出重构方案等用户确认后再动。**

---

## 阶段 3: 实施重构

1. 按方案逐步执行
2. 遵循框架规范
3. **不引入新功能、不修复旧 bug**（纯结构改动）
4. 每个原子步骤完成后自验：功能行为不变

---

## 阶段 4: 自查

读取 `references/code-reviewer.md`，使用 **refactor profile（3 维度）**：
1. 框架规范合规
2. 组件复用（重构后复用度是否提升）
3. 安全性（确认重构没引入新问题、没改变外部行为）

**发现问题 → 立即修复。最多 1 轮。**

---

## 阶段 5: Playwright 功能验证与 UI 截图对比（可选）

重构完成后，通过 Playwright 验证功能等价性并进行 UI 截图对比，确保重构未改变外部行为和视觉表现。

> **本阶段为可选步骤。** 如果项目未安装 Playwright 或无法启动页面，可跳过并继续后续阶段。

### 5.1 前置条件检查

1. **读取环境缓存** — 检查 `.harness-env.json` 中的 `playwright.installed` 字段
   - 如果 `installed: true` → 直接进入功能验证
   - 如果 `installed: false` → 提示用户可通过 `npm init playwright@latest` 安装，跳过本阶段
   - 如果字段不存在 → 执行实时检测（`npx playwright --version`），并将结果回写到 `.harness-env.json`
2. 检查项目是否有已有的 Playwright 测试配置（如 `playwright.config.ts`）
3. 确认开发服务器可启动并访问目标页面
   - 无法启动 → 跳过本阶段

### 5.2 功能验证

针对本次重构涉及的页面/组件，编写或运行 Playwright 测试：
1. 功能等价性验证 — 重构前后页面行为完全一致
2. 页面可访问性 — 目标页面能正常加载，无控制台报错
3. 关键交互无回归 — 所有交互行为与重构前一致
4. 如果项目已有 e2e 测试 → 优先运行受本次重构影响的测试用例

### 5.3 UI 截图对比（推荐）

使用 Playwright 的 `page.screenshot()` API 进行截图对比：

1. **重构前基准截图**（如有条件）：在重构前对目标页面截图
   - 存入 `screenshots/before/`
2. **重构后截图**：重构完成后对同一页面截图
   - 存入 `screenshots/after/`
3. **截图对比结果**：
   - 列出每个页面的 before/after 截图路径
   - 标注是否有视觉变化
   - **重构不应引入任何视觉变化**，如有差异需重点关注

### 5.4 验证结果输出

输出验证摘要：
- 通过/失败的测试列表
- 截图对比摘要（如已执行）
- 如有失败项或视觉差异 → 标注需要关注的问题，回到阶段 3 修复后重新验证

---

## 阶段 6: 回归验证

> **注意：** 如果项目配置了自动 lint hook，lint + format 会自动执行，无需手动运行。

- lint + format 验证
- build 验证（重构可能影响构建）
- 功能等价性确认（列出验证点）

### 输出重构摘要

- **改了什么**：重构涉及的文件和改动
- **为什么改**：重构目的和收益
- **影响范围**：受影响的文件/模块
- **性能变化**（如有）：重构前后的性能对比
- **功能等价性验证**：确认重构没改变外部行为的验证点
