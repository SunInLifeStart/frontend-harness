# change: 现有功能局部修改

## 用法

```
change: <修改描述>
change: [粘贴截图] <修改描述>
```

轻量级工作流，用于修改现有功能的样式、逻辑或交互。

> **执行模式说明：** 本工作流中所有「调用 references/xxx.md」均指**读取对应文件，按其中步骤顺序执行**。所有步骤由当前 Agent 顺序完成，不启动子 Agent。

   - 存在 → 读取缓存

## 阶段 1: 理解修改意图

[阶段 1/8] 理解修改意图...

### 1.1 解析输入

- 文字描述：提取修改目标和期望效果
- 截图（如有）：分析截图内容，识别 UI 元素和问题所在
- **有截图时以截图为准**

### 1.2 定位代码

在项目中搜索相关代码：
1. 根据描述中的关键词（页面名、组件名、功能名）搜索
2. 搜索 `{{viewsDir}}`、`{{componentsDir}}` 等目录
3. 定位到具体文件和代码位置

### 1.3 分析影响范围

- 列出需要修改的文件
- 分析修改是否会影响其他组件/页面
- 评估影响范围大小

**影响 >3 个文件 → ⏸️ 先输出修改方案，等用户确认后再动**

---

## 阶段 2: 修改方案

[阶段 2/8] 修改方案...

### 2.1 加载框架规范

根据 `.harness-env.json` 中的 `framework` 字段加载对应规范：
- Vue → 读取 `references/vue-best-practices.md`
- React → 读取 `references/react-best-practices.md`

### 2.2 扫描相关资产

读取 `references/component-scanner.md`，**按需扫描**（scope: targeted）：
- 仅扫描与修改目标相关的模块：components、composables/hooks
- 不做全量扫描（路由、权限、i18n 等不相关则跳过）

### 2.3 制定修改方案

- 列出每个文件的具体改动点
- 标注是否有可复用的现有组件/函数
- 遵循最小改动原则

---

## 阶段 3: 实施修改

[阶段 3/8] 实施修改...

按方案修改代码：
1. 遵循框架规范
2. 优先复用项目已有组件/函数，没有也不过度封装
3. **最小改动原则** — 只改需要改的，不做额外"优化"
4. 有 design.md → 遵循设计规范；无 → 按 Figma/截图一比一还原
5. 如涉及新文案 → 走 i18n

---

## 阶段 4: E2E 验证（可选）

[阶段 4/8] E2E 验证...

读取 `references/phases/verify-e2e.md`，传入：
- 变更页面/组件：本次修改涉及的页面
- 验证重点：change（修改效果符合预期、关联功能无回归）

根据验证结果决定是否继续：
- ✅ 全部通过 → 继续下一阶段
- ⚠️ Playwright 未安装或无法启动 → 自动跳过，继续下一阶段
- ❌ 测试失败 → 回到阶段 3 修复后重新验证

---

## 阶段 5: 自查

[阶段 5/8] 自查...

实现完成后，按 code-reviewer 规则自查。

读取 `references/code-reviewer.md`，使用 **modify profile（4 维度）**：
1. 框架规范合规
2. 组件复用（有没有重复造轮子）
3. UI 状态完整性
4. 设计规范合规

**发现问题 → 立即修复。最多 1 轮。**

---

## 阶段 6: 代码验证（强制）

[阶段 6/8] 代码验证...

读取 `references/phases/verify-code.md`，传入：
- 工作流类型：change
- 变更文件列表：本次任务涉及的所有文件

根据验证报告的总判定决定是否继续：
- ✅ 全部通过 → 继续下一阶段
- ⚠️ 验证不完整 → 继续，但在最终输出中标注
- ❌ 未通过 → 修复后重试，连续 3 次失败 → 报告失败详情并停止

---

## 阶段 7: 测试验证

[阶段 7/8] 测试验证...

读取 `references/phases/verify-test.md`，传入：
- 测试范围：受影响范围

根据测试结果决定是否继续：
- ✅ 全部通过 → 继续下一阶段
- ⚠️ 无测试文件 → 跳过，继续下一阶段
- ❌ 测试失败 → 修复后重试
