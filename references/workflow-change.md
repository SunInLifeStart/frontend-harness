# change: 现有功能局部修改

## 用法

```
change: <修改描述>
change: [粘贴截图] <修改描述>
```

轻量级工作流，用于修改现有功能的样式、逻辑或交互。

> **执行模式说明：** 本工作流中所有「调用 references/xxx.md」均指**读取对应文件，按其中步骤顺序执行**。所有步骤由当前 Agent 顺序完成，不启动子 Agent。

---

## 阶段 0: 环境预检

1. **读取环境配置** — 检查 `.harness-env.json` 是否存在：
   - 不存在 → 调用 `references/framework-detector.md` 执行环境探测
   - 存在 → 读取缓存

---

## 阶段 1: 理解修改意图

### 1.1 解析输入

- 文字描述：提取修改目标和期望效果
- 截图（如有）：分析截图内容，识别 UI 元素和问题所在
- **有截图时以截图为准**

### 1.2 定位代码

在项目中搜索相关代码：
1. 根据描述中的关键词（页面名、组件名、功能名）搜索
2. 搜索 `{{viewsDir}}`、`{{componentsDir}}` 等目录
3. 定位到具体文件和代码位置

### 1.3 分析影响范围

- 列出需要修改的文件
- 分析修改是否会影响其他组件/页面
- 评估影响范围大小

**影响 >3 个文件 → ⏸️ 先输出修改方案，等用户确认后再动**

---

## 阶段 2: 修改方案

### 2.1 加载框架规范

根据 `.harness-env.json` 中的 `framework` 字段加载对应规范：
- Vue → 读取 `references/vue-best-practices.md`
- React → 读取 `references/react-best-practices.md`

### 2.2 扫描相关资产

读取 `references/component-scanner.md`，**按需扫描**（scope: targeted）：
- 仅扫描与修改目标相关的模块：components、composables/hooks
- 不做全量扫描（路由、权限、i18n 等不相关则跳过）

### 2.3 制定修改方案

- 列出每个文件的具体改动点
- 标注是否有可复用的现有组件/函数
- 遵循最小改动原则

---

## 阶段 3: 实施修改

按方案修改代码：
1. 遵循框架规范
2. 优先复用项目已有组件/函数，没有也不过度封装
3. **最小改动原则** — 只改需要改的，不做额外"优化"
4. 有 design.md → 遵循设计规范；无 → 按 Figma/截图一比一还原
5. 如涉及新文案 → 走 i18n

---

## 阶段 4: Playwright 功能验证与 UI 截图对比（可选）

修改完成后，通过 Playwright 验证修改效果并进行 UI 截图对比，确保修改符合预期且未引入视觉回归。

> **本阶段为可选步骤。** 如果项目未安装 Playwright 或无法启动页面，可跳过并继续后续阶段。

### 4.1 前置条件检查

1. **读取环境缓存** — 检查 `.harness-env.json` 中的 `playwright.installed` 字段
   - 如果 `installed: true` → 直接进入功能验证
   - 如果 `installed: false` → 提示用户可通过 `npm init playwright@latest` 安装，跳过本阶段
   - 如果字段不存在 → 执行实时检测（`npx playwright --version`），并将结果回写到 `.harness-env.json`
2. 检查项目是否有已有的 Playwright 测试配置（如 `playwright.config.ts`）
3. 确认开发服务器可启动并访问目标页面
   - 无法启动 → 跳过本阶段

### 4.2 功能验证

针对本次修改涉及的页面/组件，编写或运行 Playwright 测试：
1. 页面可访问性 — 目标页面能正常加载，无控制台报错
2. 修改效果验证 — 修改的样式、逻辑或交互符合预期效果
3. 关联功能无回归 — 修改涉及的相关功能仍正常工作
4. 如果项目已有 e2e 测试 → 优先运行受本次修改影响的测试用例

### 4.3 UI 截图对比（推荐）

使用 Playwright 的 `page.screenshot()` API 进行截图对比：

1. **修改前基准截图**（如有条件）：在代码修改前对目标页面截图
   - 存入 `screenshots/before/`
2. **修改后截图**：代码修改完成后对同一页面截图
   - 存入 `screenshots/after/`
3. **截图对比结果**：
   - 列出每个页面的 before/after 截图路径
   - 标注是否有视觉变化
   - 视觉变化是否符合修改预期

### 4.4 验证结果输出

输出验证摘要：
- 通过/失败的测试列表
- 截图对比摘要（如已执行）
- 如有失败项 → 标注需要关注的问题，回到阶段 3 修复后重新验证

---

## 阶段 5: 自查

实现完成后，按 code-reviewer 规则自查。

读取 `references/code-reviewer.md`，使用 **modify profile（4 维度）**：
1. 框架规范合规
2. 组件复用（有没有重复造轮子）
3. UI 状态完整性
4. 设计规范合规

**发现问题 → 立即修复。最多 1 轮。**

---

## 阶段 6: 验证

> **注意：** 如果项目配置了自动 lint hook，lint + format 会自动执行，无需手动运行。

### 6.1 测试验证

读取 `references/test-runner.md`：
- 检查是否有与修改相关的测试文件
- 有 → 按步骤验证
- 无 → 跳过

### 6.2 变更摘要

输出：
- 修改了哪些文件
- 每个文件改了什么
- 影响范围评估
