# test-runner — 测试执行

## 用途

执行测试验证。两种模式：
1. **Markdown 测试用例** → 生成检查清单（checklist），**不给出 PASS/FAIL 判定**
2. **自动化测试文件** → 实际运行并报告结果

## 输入

- 测试文件路径
- 可选：关联的源代码文件路径

## 输出

检查清单或测试运行结果

---

## 模式 A: Markdown 测试用例（test-cases.md）

**重要：这是静态分析，不是实际执行。输出为检查清单，需人工验证。**

```
1. 读取 test-cases.md，解析所有测试用例
2. 对每个用例：
   ├── 读取「前置条件」→ 检查代码中是否有对应状态处理
   ├── 读取「操作步骤」→ 追踪代码中的事件处理链路
   ├── 读取「预期结果」→ 检查代码逻辑是否能产生预期输出
   └── 生成检查项 + 代码位置引用
3. 输出检查清单
```

**输出格式：**
```markdown
## 测试检查清单（⚠️ 静态分析结果，需人工验证）

| # | 用例 | 代码覆盖 | 关键代码位置 | 需人工确认 |
|---|------|----------|-------------|-----------|
| 1 | 正常提交流程 | ✅ 逻辑路径存在 | Form.vue:42-58 | 实际提交行为 |
| 2 | 空数据展示 | ✅ 有空态处理 | List.vue:15 | 空态 UI |
| 3 | 接口报错处理 | ⚠️ catch 块存在但无 UI 反馈 | api.js:23 | 错误提示 |
| 4 | 超长文本输入 | ❌ 未找到长度限制 | - | 需补充处理 |
```

**建议：** 如果用例较多，建议生成可执行测试文件（Vitest/Jest），放在 test/ 目录下实际运行。

### 生成可执行测试（可选）

当 test-cases.md 包含 3 个以上用例时，建议同时生成对应的可执行测试文件：
- Vue 项目 → `demand/<需求名>/test/<module>.spec.js`（Vitest）
- React 项目 → `demand/<需求名>/test/<module>.test.tsx`（Jest）
- 然后按模式 B 实际运行

---

## 模式 B: 自动化测试文件

```
1. 检测测试框架（从 .harness-env.json 读取 testFramework）：
   ├── vitest → npx vitest run <file> --reporter=verbose
   ├── jest → npx jest <file> --verbose
   └── none → 输出提示，跳过
2. 解析运行结果：
   ├── 通过的用例列表
   ├── 失败的用例 + 错误信息 + 堆栈
   └── 总计：X passed, Y failed, Z skipped
```

---

## 无测试文件

如果目标路径下没有任何测试文件：
- 输出：「无测试用例，跳过测试阶段。」
- 不中断工作流
