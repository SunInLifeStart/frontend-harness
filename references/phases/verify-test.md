# 测试验证阶段（Phase: Verify Test）

> 代码修改后的测试验证阶段。
> 工作流通过「读取 references/phases/verify-test.md 执行」调用。

## 输入
- 测试范围：`全量` | `受影响范围` | `最小` | `性能相关`
  - **全量**（refactor）：运行项目全部测试，确保重构未破坏任何功能
  - **受影响范围**（feat/bug/change）：运行与本次变更相关的测试文件
  - **最小**（hotfix）：仅运行直接相关的测试，优先速度
  - **性能相关**（perf）：运行性能相关测试，确认优化未破坏功能
- 可选：`demand/<需求名>/test/test-cases.md` 路径（如适用）

## 执行

读取 `references/test-runner.md`，按步骤执行：

### 1. 检查测试用例文档
- 如有 `test-cases.md` → 按模式 A 执行静态分析，生成检查清单
- 如无 → 跳过此步

### 2. 检查自动化测试文件
- 根据测试范围参数确定测试文件集合：
  - `全量` → 运行所有测试文件
  - `受影响范围` → 根据变更文件列表推断受影响的测试文件
  - `最小` → 仅运行与修改文件直接对应的测试
  - `性能相关` → 运行包含性能基准的测试文件
- 有测试文件 → 按模式 B 实际运行并报告结果
- 无测试文件 → 输出「无测试用例，跳过测试阶段。」

## 输出

根据执行的模式输出对应结果：
- **模式 A**：STATIC CHECK — 测试检查清单（Markdown 表格），需人工验证
- **模式 B**：PASS / FAIL — 实际测试运行结果（通过数、失败数、跳过数）
- **无测试**：跳过提示，不中断工作流

## 失败处理
- 测试失败 → 输出失败用例和错误详情，由调用方决定是否修复重试
