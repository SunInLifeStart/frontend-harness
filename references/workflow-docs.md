# docs: 文档更新

## 用法

```
docs: <修改描述>
```

轻量级文档更新工作流，用于修改 README、注释、JSDoc、CHANGELOG 等文档内容。

> **执行模式说明：** 本工作流中所有步骤由当前 Agent 顺序完成，不启动子 Agent。

---

## 阶段 0: 确认修改范围

### 0.1 解析修改意图

- 理解用户要修改哪些文档
- 确认修改类型：README / 代码注释 / JSDoc / CHANGELOG / 其他文档

### 0.2 定位目标文件

根据修改意图定位需要修改的文件：
- README → 项目根目录或子包目录
- 代码注释 / JSDoc → 搜索目标函数/组件所在文件
- CHANGELOG → 项目根目录
- 其他文档 → 按描述搜索

### 0.3 确认范围

- 列出需要修改的文件清单
- 预估修改量

---

## 阶段 1: 执行文档修改

按修改类型执行：

### README / 通用文档

- 遵循项目已有的文档格式和风格
- Markdown 格式规范（标题层级、列表风格、代码块语言标注）
- 保持与项目其他文档的一致性

### 代码注释

- 注释使用**中文**（遵循项目编码规约）
- 解释 why，不解释 what
- 复杂业务逻辑必须有注释
- 删除过时 / 误导性注释

### JSDoc / TSDoc

```typescript
/**
 * 获取用户列表
 *
 * @param params - 查询参数
 * @param params.page - 页码
 * @param params.pageSize - 每页条数
 * @returns 用户列表和分页信息
 */
```

- 公开 API 必须有 JSDoc
- 参数和返回值类型与 TypeScript 类型一致
- 示例代码（如需要）

### CHANGELOG

- 遵循 Keep a Changelog 格式
- 按 Added / Changed / Fixed / Removed 分类
- 关联 issue 或 PR 编号（如有）

---

## 阶段 2: 检查

### 2.1 拼写检查

- 检查中文文案有无错别字
- 检查英文单词拼写
- 检查技术术语拼写（API 名、库名、命令等）

### 2.2 链接验证

- 检查文档中的链接格式正确
- 检查相对路径引用有效
- 检查锚点引用存在

### 2.3 格式一致性

- Markdown 格式规范（标题层级连续、列表缩进统一）
- 代码块有语言标注
- 与项目其他文档风格一致

---

## 阶段 3: 完成

输出：
- **修改了哪些文件**
- **每个文件改了什么**
- **修改原因**
