# bug: Bug 修复

## 用法

```
bug: <bug 描述>
bug: [粘贴截图] <bug 描述>
```

精准定位 + 最小修复，不重构，不做额外优化。

> **执行模式说明：** 本工作流中所有「调用 references/xxx.md」均指**读取对应文件，按其中步骤顺序执行**。所有步骤由当前 Agent 顺序完成，不启动子 Agent。

---

## 阶段 0: 环境预检

1. **读取环境配置** — 检查 `.harness-env.json` 是否存在：
   - 不存在 → 调用 `references/framework-detector.md` 执行环境探测
   - 存在 → 读取缓存

---

## 阶段 1: 问题诊断

### 1.1 解析 Bug 描述

- 文字描述：提取现象、复现条件、预期行为
- 截图（如有）：分析 UI 异常、报错信息、样式问题
- **有截图时以截图为准**

### 1.2 定位问题代码

1. 根据 bug 描述中的页面/功能关键词搜索代码
2. 搜索 `{{viewsDir}}`、`{{componentsDir}}` 定位页面组件
3. 追踪事件处理链、API 调用链
4. 检查相关 store、composable、util 函数

### 1.3 根因分析

**必须分析根因，不能只处理表象。**

常见前端 bug 根因：
- 数据层：API 返回值结构不匹配、空值未处理、类型错误
- 状态层：响应式丢失、state 更新时序错误、computed 依赖遗漏
- UI 层：CSS 选择器优先级、布局塌陷、z-index 冲突、响应式断点
- 交互层：事件绑定错误、防抖/节流问题、异步竞态
- 权限层：路由守卫遗漏、权限配置缺失

### 1.4 信息不足

如果信息不够定位问题：
- 询问用户：复现步骤、浏览器、设备、报错信息
- 不猜测

---

## 阶段 2: 修复方案

### 2.1 加载框架规范

根据 `.harness-env.json` 中的 `framework` 字段加载对应规范：
- Vue → 读取 `references/vue-best-practices.md`
- React → 读取 `references/react-best-practices.md`

### 2.2 设计最小修复

- **只修 bug 本身**，不重构周边代码
- **不做"顺手优化"**
- 如果修复可能影响其他功能 → ⏸️ 告知用户风险，等确认

---

## 阶段 3: 实施修复 + 自查

### 3.1 实施修复

按方案实施最小修复：
1. 遵循框架规范
2. 不改任何不相关的代码
3. 有 design.md → 修复后的 UI 符合设计规范

### 3.2 自查

读取 `references/code-reviewer.md`，使用 **fix profile（2 维度）**：
1. 框架规范合规
2. 安全性（确认修复没引入新问题）

**发现问题 → 立即修复。最多 1 轮。**

---

## 阶段 4: Playwright 功能验证与 UI 截图对比（可选）

修复完成后，通过 Playwright 验证修复效果并进行 UI 截图对比，确保 bug 已修复且未引入新的视觉问题。

> **本阶段为可选步骤。** 如果项目未安装 Playwright 或无法启动页面，可跳过并继续后续阶段。

### 4.1 前置条件检查

1. **读取环境缓存** — 检查 `.harness-env.json` 中的 `playwright.installed` 字段
   - 如果 `installed: true` → 直接进入功能验证
   - 如果 `installed: false` → 提示用户可通过 `npm init playwright@latest` 安装，跳过本阶段
   - 如果字段不存在 → 执行实时检测（`npx playwright --version`），并将结果回写到 `.harness-env.json`
2. 检查项目是否有已有的 Playwright 测试配置（如 `playwright.config.ts`）
3. 确认开发服务器可启动并访问目标页面
   - 无法启动 → 跳过本阶段

### 4.2 功能验证

针对本次 bug 修复涉及的页面/组件，编写或运行 Playwright 测试：
1. 复现场景验证 — 模拟原始 bug 的复现步骤，确认问题已修复
2. 页面可访问性 — 目标页面能正常加载，无控制台报错
3. 关联功能无回归 — 修复涉及的相关功能仍正常工作
4. 如果项目已有 e2e 测试 → 优先运行受本次修复影响的测试用例

### 4.3 UI 截图对比（推荐）

使用 Playwright 的 `page.screenshot()` API 进行截图对比：

1. **修复前基准截图**（如有条件）：在代码修复前对目标页面截图
   - 存入 `screenshots/before/`
2. **修复后截图**：代码修复完成后对同一页面截图
   - 存入 `screenshots/after/`
3. **截图对比结果**：
   - 列出每个页面的 before/after 截图路径
   - 标注是否有视觉变化
   - 视觉变化是否仅为修复预期内的改动

### 4.4 验证结果输出

输出验证摘要：
- 通过/失败的测试列表
- 截图对比摘要（如已执行）
- 如有失败项 → 标注需要关注的问题，回到阶段 3 修复后重新验证

---

## 阶段 5: 回归验证

> **注意：** 如果项目配置了自动 lint hook，lint + format 会自动执行，无需手动运行。

### 5.1 Bug 验证

从代码逻辑层面验证：
- bug 根因已消除
- 相关的 UI 状态处理正确
- 没有引入新的问题

### 5.2 修复摘要

输出：
- **根因**：bug 是因为什么导致的
- **修复方式**：改了什么、怎么改的
- **影响范围**：修复影响了哪些文件/功能
- **回归风险**：有没有可能影响其他功能
